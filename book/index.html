Procrastinate: Task Queues on Postgres

Procrastinate is a Python task queue with an unusual opinion: the message broker you already run is called PostgreSQL. This course takes you from "what is a background job?" to operating workers in production — retries, locks, scheduling, graceful shutdown, and a real deployment on AWS ECS. Every example in the book was run against a real Postgres database, every claim about a default or a gotcha links back to the official docs, and every chapter ends with a challenge you solve by making failing tests pass.

Click Launch Codespace in the sidebar for a ready-to-go environment, or clone the repo and run uv sync && docker compose up -d locally.

Part I · Foundations

  1. 1Why a task queue — and why PostgresWhat background jobs actually are, and how a database becomes a broker with SELECT FOR UPDATE and LISTEN/NOTIFY.
  2. 2Setup: your first job, end to endInstall, connect, apply the schema, defer one job, and watch a worker pick it up.
  3. 3Defining and deferring tasksTask anatomy, configure(), batch deferring, JSON argument limits, and blueprints for larger codebases.
  4. 4Workers up closeQueues, concurrency, how jobs are fetched the instant they're deferred — and the polling that backs it up.

Part II · Controlling execution

  1. 5Retries and failureThe job lifecycle, built-in retry strategies, and writing a custom one that changes queue or priority on retry.
  2. 6Scheduling: later and every hourOne-off future jobs with schedule_at/schedule_in, and periodic tasks with cron — including who defers them when you run five workers.
  3. 7Locks, queueing locks, prioritiesSerialize jobs that touch the same resource, stop duplicate jobs from piling up, and decide who goes first.
  4. 8Context, cancellation, middlewareReach into the running job with JobContext, abort jobs that are already executing, and wrap every task with middleware.

Part III · Sync and the web

  1. 9Sync and async togetherSync connectors, deferring from synchronous code, and the RuntimeError waiting for you at the boundary.
  2. 10Procrastinate in FastAPIDefer jobs from endpoints, embed a worker in the lifespan — and why production usually wants a separate worker process.

Part IV · Production

  1. 11Testing your tasksInMemoryConnector, asserting on deferred jobs, and running a real worker inside a test when it's worth it.
  2. 12Operating in productionSchema migrations without downtime, monitoring, graceful shutdown, stalled-job recovery, and keeping the jobs table small.
  3. 13Deploying workers on AWS ECSContainerize the worker, map SIGTERM to procrastinate's graceful shutdown, and size stopTimeout so jobs survive deploys.
