Chapter 2 · Part I — Foundations

Setup: your first job, end to end

This chapter assembles the four pieces every Procrastinate project has: a connector that knows how to reach Postgres, an App that ties tasks to that connector, a one-time schema applied to the database, and the worker loop. By the end you'll have run the whole cycle and know where two of the most common setup traps hide.

Install and connect

Procrastinate installs with pip install procrastinate (this course's repo uses uv sync, which reads the same dependency from pyproject.toml). It requires Python 3.10+ and PostgreSQL 13+, per the project's own description. The psycopg driver comes along as a dependency. The course repo also ships a compose.yaml that starts Postgres 17 on port 5441, so the examples never collide with a Postgres you already run.

The App is Procrastinate's entry point: it holds the task registry and the connection. You hand it a connector, the object that owns the actual database pool. There are five connector classes; the two that matter for now are PsycopgConnector (async, built on psycopg 3) and its synchronous twin SyncPsycopgConnector. Workers require the async one, which is why it's the default choice for this whole course; chapter 9 covers when the sync classes earn their keep.

Three details in that file do more work than they look like they're doing. The connector accepts a conninfo DSN string, but it could equally take keyword arguments (kwargs={"host": ..., "user": ...}) or nothing at all, in which case psycopg falls back to the standard PGHOST/PGUSER-style libpq environment variables, the form you'll want in containers. app.open_async() opens the connection pool, and the async with form closes it automatically on the way out. And run_worker_async(wait=False) runs a real worker that exits once the queue is empty instead of waiting forever. That's handy in scripts and, as chapter 11 shows, in tests.

⚠ Gotcha

procrastinate: error: Could not load app from examples.hello_world.app — if you see this, the CLI simply can't import your module, because unlike python -m, a console script doesn't put the current directory on sys.path. The fix is the export PYTHONPATH=. above, which the official quickstart itself calls out. We hit it building this course, on the first CLI command we ran.

One deferred job, executed by a worker in the same process, with its final status read back from the database. That's the entire loop the next eleven chapters elaborate on. But notice the warning on the first line; it's worth taking seriously.

Where the app object should live

Procrastinate warns because our App was created in the module Python ran directly (__main__). For a demo script that's harmless. In a real project it sets up a genuinely nasty failure, described in the docs' top-level-app discussion: if the module defining your app is run as __main__ and your tasks live in another module that imports it, Python imports the app's module a second time under its real name, and "you will end up with two distinct instances of your app": one holding your task registry, one actually connected, neither complete.

You can see the seed of the problem in this chapter's own artifacts. The worker log in chapter 1 and the shell output below both record task names like __main__.greet: a task's default name is derived from the module it was defined in, and our module was __main__. A separate worker process importing the same file would know that task as examples.hello_world.greet instead, and jobs deferred under one name aren't found under the other. The recommended layout is a dedicated module for the app (say myproject/procrastinate_app.py), with import_paths=["myproject.tasks"] pointing at the modules that define tasks. The course examples keep app creation inside examples/common.py for the same reason.
