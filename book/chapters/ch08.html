Chapter 8 · Part II — Controlling execution

Context, cancellation, middleware

Three tools for the space between "the job runs" and "the job runs the way production needs it to." JobContext lets a task see its own execution: which job, which worker, which attempt. Cancellation stops work, including work already in flight. And middleware wraps every task with the logging, timing, and tracing you don't want copy-pasted into each one.

JobContext: the job, from the inside

Add pass_context=True to the decorator and the task's first argument becomes a JobContext: the full job row (id, queue, attempts, kwargs), the worker's name and queues, and a should_abort() method we'll need shortly.

The context also carries additional_context: a dict you pass to run_worker_async(additional_context={...}), useful for handing every task a shared HTTP session or database handle owned by the worker. The docs are explicit about its limits: it's one mutable dict shared by all concurrent tasks, it isn't persisted anywhere, and it "is not supposed to be used for passing data from a task to a future task." For task-to-task data, use the database you're already sitting on.

Stopping jobs

A job that hasn't started is easy: app.job_manager.cancel_job_by_id_async(job_id) moves a todo row to cancelled (add delete_job=True to remove the row entirely). A job that's already running is more interesting. Passing abort=True flags the job, and the worker learns about it through the same two-channel design as job fetching: a Postgres notification, backed by polling (abort_job_polling_interval, 5 seconds by default in the installed 3.9.0).

What happens next depends on the task's flavor, per the cancellation guide. An async task is cancelled through asyncio: CancelledError is raised at its current await point, and if some section must not be interrupted, you shield it and re-raise. A sync task can't be interrupted from outside at all; it has to cooperate by checking context.should_abort() periodically and raising JobAborted. Either way the row ends up aborted:

⚠ Gotcha

Abortion trumps retry policy: "when a job is requested to abort and that job fails, it will not be retried (regardless of the retry strategy)." Sensible (the human asked it to stop), but surprising if you expected your retry= to resurrect it.

Source: Procrastinate docs — cancellation

Middleware: wrap every task once

New in the 3.x line, middleware is a callable that wraps task execution, with the signature (call_next, context, worker). Call call_next() to run the next layer (ultimately the task), and do your before/after work around it:

There are two registration points and one matching rule, laid out in the middleware guide. Task middleware runs in the task's own context, so it must match the task's flavor: async def middleware for async tasks, plain def for sync tasks (which run in threads). Worker middleware is always async and wraps every job regardless of flavor. The chain runs worker middleware, then worker-wide task middleware, then per-task middleware (via @app.task(task_middleware=[...])), then the task. And one rule matters more than the rest: middleware must let exceptions propagate. Chapter 5's retries and the job's failure status both depend on the exception reaching the worker; try/finally is your friend, except: pass is sabotage.
