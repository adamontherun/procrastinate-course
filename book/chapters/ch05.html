Chapter 5 · Part II — Controlling execution

Retries and failure

The email provider times out. The third-party API returns a 503. The database deadlocks once a week. Background jobs exist partly because this is normal, and a task queue earns its keep by what it does next. In Procrastinate, "what happens next" is a retry strategy: a policy object attached to the task, consulted every time it fails.

What a failure actually is

When a task raises, the worker catches the exception, logs it, and asks the task's retry strategy for a decision. No strategy (the default) means the job's row moves to failed and stays there. With a strategy, the job can instead go back to todo, possibly scheduled for later, and chapter 1's state machine gains its loop: a retryable failure is just a transition back to the start. The difference is visible in the events Procrastinate records: deferred_for_retry versus failed, in the events table.

The built-in options

The retry= parameter on the task decorator accepts three shapes, per the retry guide. retry=True retries forever, which is almost never what you want. retry=5 retries five times, so six executions in total. And a RetryStrategy object gives you the real controls:

RetryStrategy takes wait (a constant delay in seconds between attempts), linear_wait, or exponential_wait for growing backoff, and retry_exceptions, a set of exception types that are worth retrying. That last one encodes a distinction worth internalizing: a ConnectionError might succeed next time, but a ValueError from a malformed argument will fail every time you run it, and retrying it five times with exponential backoff just delays the inevitable while hiding the bug.

⚠ Gotcha

Despite its name, max_attempts counts retries, not executions: the source documents it as "the maximum number of attempts the job should be retried," and stops when the count of previous attempts reaches it. So max_attempts=3 allows four executions. We hit this building this chapter's challenge: a test expecting three executions saw a fourth. If you need "at most N runs," write max_attempts=N-1.

Writing your own strategy

For anything the built-in knobs can't express, subclass BaseRetryStrategy and implement get_retry_decision(). It receives the exception and the job, and returns either None (give up, the job fails) or a RetryDecision, which can set when to retry (retry_in/retry_at) and also move the job: a new priority, queue, or lock. One escape hatch to know about before chapter 12 uses it: a job that has already failed isn't gone. The same guide shows app.job_manager.retry_job_by_id_async() for programmatic resurrection, and chapter 2's admin shell has a retry command for doing it by hand.

🏭 In production

Retries with backoff interact with everything around them: a job retried in an hour holds any lock context it was configured with (chapter 7's caveats apply), and a worker deploy that changes task code means a retried job runs the new code with the old arguments. Keeping task arguments small and stable (ids, not payloads) is what makes that safe.
