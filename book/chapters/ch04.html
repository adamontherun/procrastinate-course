Chapter 4 · Part I — Foundations

Workers up close

A worker is a loop: fetch a job, run it, record the result, repeat. The interesting engineering lives in the adverbs. How does it fetch a job the moment one appears? How does it run four jobs at once in one process? Which queues does it watch? This chapter answers those with measured timings from the course database and a walk through a real worker's log.

Starting one

In production a worker is usually its own process, started with the CLI: procrastinate --app=path.to.app worker, plus queue names if you want a subset. From code, the worker guide offers two entry points that differ in who owns the lifecycle: app.run_worker() is self-contained (it starts an event loop, opens the app, and closes both when done), while await app.run_worker_async() expects the app to already be open; that's the embeddable form chapter 10 threads into FastAPI. Every example so far has used the pattern worth committing to memory for scripts and tests: wait=False, which stops the worker when the queue runs dry instead of idling for more.

Queues route work; concurrency overlaps it

A worker given queues=["slow"] only ever fetches jobs whose queue column says slow; a worker given nothing listens to every queue. Queues are how you keep a pile of bulk exports from delaying password-reset emails: run one worker per class of work, sized to it. Queues appear the first time something uses them: there's nothing to declare or create.

Concurrency is the other axis. By default a worker runs one job at a time (concurrency=1 in the worker's signature). Setting it higher runs that many async sub-workers on one event loop, which pays off exactly when tasks spend their time awaiting I/O: "while a job is waiting for external I/O (usually network), the CPU is free to run other jobs," as the concurrency guide puts it. Serial execution costs the sum (4.1 seconds); four sub-workers cost roughly the longest job (1.0 second). And the reports job sat untouched at todo through both runs because neither worker was listening to its queue. Routing is absolute, not advisory.

⚠ Gotcha

Concurrency > 1 buys nothing if the task blocks the event loop. A task that calls time.sleep(1), or does heavy computation, or uses a blocking HTTP client like requests, freezes all sub-workers for that second. Blocking work belongs in sync tasks, which run in threads (chapter 9), or in more worker processes rather than more sub-workers.

How fetching actually works

Chapter 1 previewed the mechanism; the worker log makes it concrete. Here's a real session: defer a job with the CLI, then run a verbose worker for five seconds.

Reading top to bottom: the worker registers itself in the database (a row with a heartbeat, which chapter 12 uses to detect dead workers), installs signal handlers so Ctrl-C shuts it down cleanly, checks for periodic tasks (chapter 6), runs our job, and settles into its waiting rhythm. That waiting is a belt-and-suspenders design: the worker LISTENs for the notification Postgres sends when a job is deferred, so pickup is effectively immediate, and it also polls every 5 seconds by default (fetch_job_polling_interval=5.0, verified against the installed 3.9.0 source) in case a notification was missed. The discussions page describes exactly this: notifications for latency, polling "just in case."

🏭 In production

The LISTEN/NOTIFY connection is a real, dedicated database connection per worker, on top of the pool (10 connections by default, the psycopg_pool default). On a big fleet, or behind a connection pooler that doesn't support LISTEN, you can trade latency for connections: listen_notify=False disables the notification channel and leaves fetching to the polling interval alone.

Worker options you'll actually set

The full option list lives on App.run_worker's API page; beyond queues, concurrency, and wait, two more come up repeatedly in this course. name= labels the worker in logs and in its database registration, which matters as soon as you run more than one. install_signal_handlers=False tells the worker not to claim SIGINT/SIGTERM for itself, so it can live inside a larger application that has its own shutdown story; that's chapter 10's FastAPI embedding. Rather than passing these at every call site, an app can set worker_defaults={...} once at construction.
