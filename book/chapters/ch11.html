Chapter 11 · Part IV — Production

Testing your tasks

Task code has two things worth testing, and they're different tests. Does your application defer the right jobs with the right arguments? And does the task function do the right thing when it runs? Both can be answered without a database, using a connector that keeps the "queue" in a Python dict. This chapter covers that tool, and is honest about the cases where only real Postgres will do.

The in-memory connector

procrastinate.testing.InMemoryConnector implements the connector interface against plain dictionaries, and app.replace_connector() swaps it into your existing app, as a context manager that restores the real connector afterward. Both come from the official testing guide. No schema, no connection, no cleanup between tests beyond a reset() call.

Three moves cover most test suites. Assert on in_memory.jobs to verify your code deferred what it should: each value is a dict with the task_name, the kwargs under "args", the queue, and the status. Call run_worker(wait=False, listen_notify=False) to execute everything queued, synchronously, in-process; then assert on your task's side effects. And note that the sync defer() works fine here in ordinary test functions; chapter 9's event-loop boundary only bites inside async code.

⚠ Gotcha

The in-memory worker runs jobs, but it doesn't emulate Postgres. In our runs it executed locked jobs regardless of lock semantics' subtleties, and nothing enforces the timing behaviors chapters 6 and 7 measured. If the behavior you're testing is queue semantics (ordering under locks, queueing-lock refusals, stalled detection), test against real Postgres. This course's own challenge suite does exactly that, with a fixture that truncates the job tables around each test (challenges/conftest.py).

Every test gets a fresh connector (so no reset() needed), and none of them can touch a real database by accident. The production connector needs no special handling: it was never opened, and replace_connector puts it back on exit.

🏭 In production

A useful split that has held up across this course's own suite: unit tests use InMemoryConnector and run in milliseconds with no services; a smaller integration layer runs against a throwaway Postgres (Docker makes this cheap) and covers what only Postgres decides: locks, priorities, stalled workers, schema. CI runs both; chapter 12's tooling assumes the second layer exists.

⚡ Try it

No local setup? Open this chapter in GitHub Codespaces — Postgres is already running there.

Run uv run python -m examples.ch11_testing. Notably, this is the one example in the course that works with Postgres stopped — try docker compose stop first and see.

The challenge, challenges/ch11/challenge.py, has you build the toolkit itself: a connector-swapping context manager, an assert_deferred helper, and a run-everything function. Its tests need no database either: uv run pytest challenges/ch11.
