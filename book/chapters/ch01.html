Chapter 1 · Part I — Foundations

Why a task queue — and why Postgres

Every web application eventually hits the same wall: a user clicks a button, and the work behind that click takes longer than a user is willing to stare at a spinner. Task queues exist to move that work somewhere else. This chapter builds the vocabulary the rest of the course runs on, then makes the argument Procrastinate itself makes: you may not need a message broker, because you already have PostgreSQL.

The problem background jobs solve

Suppose your checkout endpoint sends a confirmation email. The email provider's API usually answers in 200 milliseconds, occasionally in 30 seconds, and sometimes not at all. If the endpoint calls that API directly, its slowest dependency sets the user's wait time, a failed email turns into a failed checkout, and a burst of orders becomes a burst of slow requests tying up your web processes.

The task-queue answer is to split the work in two. The endpoint records an intention: "someone should send this email, with these arguments." A separate process, called a worker, picks up that recorded intention later and does the actual work. The recorded intention is a job; the function the worker runs is a task; and the place jobs wait between those two moments is the queue, which lives inside a piece of infrastructure called a broker. In Procrastinate's vocabulary, a task is a Python function you register, and a job is "the launching of a specific task with specific values for the keyword arguments." The glossary pins these terms down if you ever want the formal versions.

This split buys you three separate things. The user's request finishes as soon as the intention is recorded, not when the email lands. The work survives crashes, because the job exists independently of the process that created it. And the work can be retried, spread across machines, delayed, or rate-limited, because it's now data you can reason about instead of a stack frame in a web process.

The broker question

Celery, the best-known Python task queue, expects you to run RabbitMQ or Redis as the broker. That's an entire additional piece of stateful infrastructure: something to deploy, monitor, back up, and page someone about. Procrastinate's position, stated plainly in its design discussions, is that if you already operate PostgreSQL, "maintaining a RabbitMQ/Redis/… service is double the trouble." The queue goes in the database you already have.

Storing jobs in Postgres has a second consequence that's easy to underestimate: your queue becomes ordinary data. You can inspect stuck jobs with SELECT, count failures by task name with GROUP BY, and join jobs against your own tables. With a dedicated broker, that kind of visibility means learning its admin tooling; here it's the SQL you already know. The docs make the same point: browsing and editing jobs is easier "because interacting with data in a standard database [is] a lot easier" than with a specialized broker.

The honest counterweight is throughput. A dedicated broker exists because it's very good at exactly one thing, and a busy jobs table adds real load to a database that also serves your application. Procrastinate doesn't publish benchmark numbers, and this course won't invent any; the working assumption that carries most teams is that if your job volume is a problem for Postgres, you'll see it in the same monitoring you already point at Postgres. For the workloads most applications actually have, thousands of jobs an hour rather than millions a minute, the operational simplicity wins.

🔬 Under the hood

Two Postgres features make the whole design safe and fast. Row locking (SELECT … FOR UPDATE) guarantees that "when a worker starts executing a job, it's the only one" — two workers can't grab the same row. And LISTEN/NOTIFY means workers don't sit in a poll loop asking "anything for me?": deferring a job sends a notification that wakes listening workers immediately. Workers still poll on a timer as a backstop, "just in case" a notification is missed.

A job is a row

Everything above stays abstract until you watch it happen. The script below defers two jobs (one that will succeed, one written to fail) and reads their status out of the database before and after running a worker. Don't worry about the setup details yet; that's chapter 2. Watch the statuses.

Both jobs start as todo: rows in a table named procrastinate_jobs, holding the task's name, its arguments as JSON, and a status column. The worker moves each row to doing while it runs the function, then to succeeded or failed. Note what didn't happen when flaky_export raised: the script kept going, the other job completed, and the failure became a recorded fact you could query tomorrow. A crash in a background job is data, not a disaster.

The full state machine, from the discussions page, adds three states you'll meet in later chapters: a failed job whose retry policy allows another attempt goes back to todo (chapter 5), a job cancelled before any worker picks it up becomes cancelled, and a running job that's asked to stop becomes aborted (chapter 8).

🏭 In production

"The queue is a table" is also the escape hatch when things go wrong at 2am. Every debugging trick you know for ordinary rows (looking at them, counting them, updating one by hand in an emergency) applies to your job queue. Chapter 12 builds proper tooling for this, but the floor is already high: psql is a queue inspector.
