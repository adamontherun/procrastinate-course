Chapter 12 · Part IV — Production

Operating in production

A queue that runs for months accumulates history, survives worker crashes, and outlives several versions of the library that created it. This chapter is about that timescale: keeping the jobs table small, noticing when things break, recovering the jobs a dead worker took with it, and upgrading the schema without stopping the world. Its centerpiece is a demo where we SIGKILL a worker mid-job on purpose and watch the recovery machinery work.

Keep the table small

Finished jobs stay in procrastinate_jobs forever by default; you saw chapter 1's rows again in chapter 2's shell. There are two cleanup levers, per the cleanup guide. The blunt one is the worker option delete_jobs="always" (or "successful"), which removes each job the moment it completes:

The finer-grained lever keeps recent history for debugging and removes only old rows: the builtin task remove_old_jobs, deferred periodically with a max_hours argument. The docs' recommended shape combines chapter 6 and chapter 7: a periodic schedule plus the remove_old_jobs queueing lock so sweeps never stack up. Underneath, the builtin calls a public API you can also use directly, app.job_manager.delete_old_jobs(nb_hours=...), which is what this chapter's challenge exercises.

⚠ Gotcha

We hit a sharp edge here while writing this course's test suite: a long-known upstream bug (procrastinate #521, open since 2022, still present in 3.9.0). add_tasks_from renames a blueprint's task objects in place, so every App() constructed in a process re-registers the builtin tasks against the same shared task objects: the first app knows the task as builtin:procrastinate.builtin_tasks.remove_old_jobs, the second as builtin:builtin:…, and so on. Defer the builtin in a multi-app process (a test suite, most likely) and the job can be stamped with a name no fresh worker recognizes — or, eventually, one that overflows the 128-character name column. Real deployments construct one app per process and never notice. If your code might not, defer it by its stable dotted-string alias (app.configure_task(name="procrastinate.builtin_tasks.remove_old_jobs", ...)) or call delete_old_jobs directly.

Recover what dead workers took

Chapter 4's log showed workers registering themselves and heartbeating every 10 seconds. Here's what that buys. A worker that dies without a graceful shutdown (OOM-killed, SIGKILL, hardware) leaves its in-flight jobs stuck in doing, and chapter 7 told you the extra cost: any locks those jobs hold stay blocked. The stalled-jobs guide defines the detection rule (a job is stalled when its worker hasn't heartbeated for get_stalled_jobs' seconds_since_heartbeat, 30 seconds by default; the worker-side stalled_worker_timeout, also 30, governs pruning such workers) and the fix: find them, retry them. One caution belongs next to it: "stalled" is an inference from silence. A worker that's alive but wedged past its heartbeat looks identical, and retrying its job means the work may run twice. Once you adopt this recipe, delivery is effectively at-least-once, so tasks you let it resurrect should be idempotent.

Watching it run

You met the two CLI tools in chapter 2: healthchecks for a smoke test (configuration, connection, schema) and shell for inspecting and fixing jobs (list_jobs, retry, cancel — it also runs one-shot: procrastinate shell list_jobs). For dashboards and alerting, the queue is a table: SELECT status, COUNT(*) FROM procrastinate_jobs GROUP BY status is a metrics exporter away from a graph, and the events table adds per-transition timestamps (deferred, started, deferred_for_retry, failed, succeeded), though the docs label that API early-stage. For errors: configure the root procrastinate logger (workers emit through child loggers like procrastinate.worker), and most worker records carry an action extra naming the event, per the logging guide, and job failures are logged with the exception attached (exc_info, verified in the 3.9.0 worker source). That's exactly what Sentry-style tools ingest via their logging integrations.

Migrating the schema

The schema you applied in chapter 2 belongs to procrastinate 3.9.0; upgrading the library eventually means migrating it. Migrations ship inside the package as plain SQL files (procrastinate schema --migrations-path prints where). Since the 3.x line, filenames include the version and a pre or post marker; older legacy migrations in the same directory omit that suffix. See the migrations guide. The simple workflow stops everything: apply all migrations, upgrade the library, restart. The zero-downtime workflow interleaves per version: apply that version's pre scripts, deploy the new code, apply its post scripts, repeat. Procrastinate doesn't track which migrations have run; your migration tooling (or a checklist) does.

🏭 In production

The pre/post split exists because during a deploy, old and new library versions run against one schema at the same time. pre scripts must be compatible with the old code, post scripts only run once the old code is gone. Skipping a version in a hurry ("we'll jump from 3.6 to 3.9") means applying every intermediate version's scripts in order, not just the newest: the guide is explicit that the zero-downtime path repeats per version.
