Chapter 10 · Part III — Sync and the web

Procrastinate in FastAPI

FastAPI and Procrastinate share an event loop model, which makes the integration almost suspiciously small: open the app at startup, defer from endpoints, and optionally run a worker inside the web process itself. This chapter builds that app, verifies it over real HTTP, and then argues with its own architecture, because the embedded worker has a place, and that place is not most production systems.

Two modules, on purpose

The task queue gets its own module, exactly as chapter 2's app-placement rules demand. The web module imports from it; so could a standalone worker process later, with procrastinate --app=examples.ch10_fastapi.tasks.app worker. Nothing about the tasks knows the web framework exists.

The lifespan owns the machinery

FastAPI's lifespan context manager is the natural home for both halves of the setup: opening the connection pool, and (for the embedded flavor) starting the worker as an asyncio task. This mirrors the official FastAPI example in the worker guide, including its two load-bearing details: install_signal_handlers=False, because uvicorn owns SIGINT/SIGTERM in this process (chapter 4 previewed this), and a shutdown that cancels the worker task and waits, bounded by a timeout.

The endpoint pattern is the one to internalize regardless of where the worker lives: 202 Accepted plus a job id, not a completed result. The client got a promise, and the job id is a real handle it can poll. Here's the whole loop over actual HTTP, uvicorn on one side, curl on the other:

Embedded or separate?

The embedded worker is genuinely convenient: one process to run, one thing to deploy, and jobs start instantly in development. It's the right shape for small apps and demos. Its costs show up with scale. The worker competes with request handlers for the same event loop and CPU; scaling the web tier out multiplies workers whether or not the queue needs them; and a worker mid-job extends every web deploy's shutdown. Separating them (this same tasks.py, run by procrastinate worker in its own container) restores independent scaling and lifecycles, and chapter 13 deploys exactly that split.

⚠ Gotcha

While building this chapter we spent twenty minutes on a job that failed instantly with no traceback in the server log. The cause: a forgotten worker from an earlier chapter was still running against the same database, listening on all queues. It won the race for the new job, couldn't import examples.ch10_fastapi.tasks.send_welcome, and failed it with TaskNotFound. Any worker that can reach your database is part of your system, whether you remember it or not. If several apps must share one Postgres, give each its own schema via search_path. Separate queues are not enough, because default workers listen everywhere.

⚡ Try it

No local setup? Open this chapter in GitHub Codespaces — Postgres is already running there.

Start the app (uv run uvicorn examples.ch10_fastapi.web:api --port 8123) and replay the curl session. Then convert it to the separated architecture: delete the worker lines from the lifespan, and run PYTHONPATH=. uv run procrastinate --app=examples.ch10_fastapi.tasks.app worker in a second terminal. The curl session should behave identically.

The challenge, challenges/ch10/challenge.py: write the lifespan-shaped plumbing yourself (an async context manager that runs an embedded worker and shuts it down cleanly), plus the 202-style signup handler. uv run pytest challenges/ch10.
