Chapter 9 · Part III — Sync and the web

Sync and async together

Procrastinate is async at heart, and honest about why: a task queue has to call your code, so the worker must speak your code's dialect. But most codebases aren't purely one or the other, and the library meets them where they are: sync task functions, sync deferring, sync connectors. Each accommodation has a boundary, and this chapter maps where the boundaries actually run.

Sync tasks run in threads

You've been able to write def tasks since chapter 3; here's what that choice buys. The design discussion explains the machinery: sync tasks run through asgiref.sync.sync_to_async, each on a thread from a shared pool (concurrent tasks get separate threads; idle threads get reused), so a blocking call doesn't freeze the worker's event loop the way it would inside an async task (chapter 4's gotcha). The thread names are right there in the output:

Three time.sleep(0.5) calls, three threads, half a second total. This is the correct home for blocking HTTP clients, file I/O, and libraries that were never taught await. The flip side: threads mean the usual thread-safety rules apply to anything the tasks share, and Python's GIL still serializes CPU-bound work, so this trick parallelizes waiting, not computation.

Deferring from sync code

Every defer method has a sync twin (defer() for defer_async(), and so on), and the sync-defer guide describes what happens underneath: an async connector conjures a synchronous counterpart for those calls. But there's a boundary with a sharp edge. Sync methods on an opened async app only work from truly synchronous territory, like inside a sync job's thread. The fully synchronous program

Some programs have no event loop at all: cron scripts, WSGI apps, management commands. For those, SyncPsycopgConnector gives you a real synchronous connection pool, no asyncio anywhere:

That exception is the chapter's central rule surfacing as code. The docs state it flatly: with a synchronous connector, "the only thing you'll be able to do is defer tasks." Workers require an async connector, always (chapter 2 said so; here's the enforcement). A sync program hands work over the wall; somewhere else, an async worker process picks it up. For SQLAlchemy codebases there's even a connector that can share your engine's connection pool (SQLAlchemyPsycopg2Connector — pass your existing engine to app.open(), or it creates its own), so deferring needn't cost a second pool.

🏭 In production

This split is an architecture, not a workaround: web processes (sync or async) defer with whatever connector suits them, and a separate worker fleet with async connectors executes. The two halves share nothing but the database. Chapter 10 shows the one setup where they share a process, and chapter 13 deploys the separated version.
