Chapter 3 · Part I — Foundations

Defining and deferring tasks

A task is a decorated function; a job is one scheduled run of it. That sounds like the whole story, and for the first week of using Procrastinate it is. This chapter covers what the first week hides: how tasks get their names, what configure() can override at defer time, how to enqueue a thousand jobs in one round trip, and the one rule about arguments that bites everyone eventually.

Task anatomy

The @app.task decorator registers a function with the app. With no arguments the parentheses are optional; with arguments you set the task's defaults, most of which later chapters treat in depth: queue here and in chapter 4, retry in chapter 5, lock, queueing_lock, and priority in chapter 7, pass_context in chapter 8. Two parameters are worth settling now. First, tasks can be sync or async functions: async tasks all share the worker's event loop, while "each sync task runs in its own thread." Chapter 9 unpacks what that means for you; until then the examples stay async.

Second, name. A task's default name is its module path plus the function name (examples.hello_world.greet), and that string, not the function object, is what gets stored in the job row and looked up by workers. Renaming a module or moving a function silently renames every task in it, and jobs already in the queue still carry the old name. Setting an explicit name= decouples your queue's stable vocabulary from your codebase's layout. It isn't required, but on any project expected to survive a refactor, it's cheap insurance.

Five ways to defer

The example below defers jobs five ways; every one of them ran against the course database to produce the output that follows.

defer_async() returns the new job's integer id, the same id you saw in list_jobs and the same primary key you'd find in the table. configure() is the general mechanism behind variants 2 and 3: it returns a deferrer with some settings overridden (queue, lock, priority, scheduling, all listed in the defer guide). With task_kwargs it becomes a reusable partial application, deferred many times with the remaining arguments. Variant 4, app.configure_task(name=...), defers purely by the task's string name, which is how one service can enqueue work for another without importing its code. It's also completely unchecked: the name is only resolved when a worker tries to run the job, so a typo surfaces as a failed job, not as an exception where you deferred it.

Variant 5, batch_defer_async(), inserts all the jobs in a single database operation, which is the difference between one round trip and a thousand when a nightly fan-out enqueues per-user work. One caveat from the batch-defer guide to file away for chapter 7: don't put two jobs sharing a queueing lock in the same batch, because the resulting AlreadyEnqueued error rolls back the entire batch.

Arguments travel as JSON

Job arguments are serialized with json.dumps into the job row, which imposes the JSON type system on everything you pass: strings, numbers, booleans, None, lists, dicts. The example deliberately passes a datetime.date, and the defer call itself, not the worker later, raises TypeError: Object of type date is not JSON serializable. Failing at defer time is the good version of this problem; you're still in the code that knows what it meant.

⚠ Gotcha

The subtler version is types JSON can represent but doesn't round-trip: a tuple arrives as a list, and dict keys that were integers arrive as strings. The worker sees whatever json.loads produces, not what you passed. If you need richer types, the connector accepts custom json_dumps/json_loads functions; the pragmatic default is to pass ids and look the objects up in the task, which also avoids stale data when the job runs an hour later.

Blueprints: tasks without an app

The decorator pattern has a structural weakness: @app.task needs the app at import time, so every module of tasks imports your app module, and reusable packages can't ship tasks at all without knowing about your project. A Blueprint is the same registry without the app:

The app adopts it with app.add_tasks_from(reports, namespace="reports"). The namespace prefixes every task name, preventing collisions between packages. The exact format is worth knowing because you'll see it in job rows and need it for defer-by-name: the namespace, a colon, then the task's original dotted name. That's why variant 4 above used reports:examples.ch03_blueprint_tasks.build. Our first draft guessed reports.build, and the worker failed the job with TaskNotFound. An explicit name= on the blueprint task would have made this predictable: @reports.task(name="build") yields reports:build.
